# 2026-07-20 — Стартовые наборы («касты»)

## Что сделано

Введена система стартовых наборов (starter kits, «касты») — разнообразие
стартового снаряжения вместо единого набора. Требование владельца, детали
согласованы в чате и записаны в `docs/07-decisions.md` §13.

### Дизайн (со слов владельца)
- Выдача — **выбор в лобби через GUI**; в меню всегда «Без касты» и «Случайная».
- Каст = **доп. предметы + своё стартовое золото** (перекрывает `start-gold`).
  Системные предметы (вилка/помощник/блок возрождения) — всегда.
- **Всё снаряжение почти сломано** (`damage` ~85–90%): двигатель игры — ремонт.
  Золото минимальное (4–12). Разница кастов — в РОЛИ, не в силе (бомжеватый §11).
- Глобальная библиотека-шаблон + **копии на арену** (правишь копию, не оригинал).

### Код
Новые файлы:
- `kit/Kit.java` — модель: name/icon/gold/lore/items. `load` читает и рукописный
  формат (`Items.fromSpec`), и сериализованные ItemStack (копия арены после GUI).
  `apply(Player)` — броня авто-надевается в слоты, остальное в инвентарь.
  `copy(newId)` — глубокая копия под копирование на арену.
- `kit/KitRegistry.java` — глобальный `kits.yml` (read-only, `saveResource` при
  отсутствии). Оригинал командами не пишется → рукописный формат и комментарии целы.
- `menu/KitSelectMenu.java` — выбор в лобби (сетка 54, касты + «Без касты» barrier
  + «Случайная» ender_pearl), выбранное подсвечивается `setEnchantmentGlintOverride`.
- `menu/KitEditorMenu.java` — админ-редактор предметов каста (как `LootEditorMenu`:
  разложил → пересобралось на закрытии).
- `resources/kits.yml` — 5 стандартных: warrior/archer/angler/miner/dealer.

Правки:
- `arena/Arena.java` — список кастов + `default-kit`, load/save `arenas/<id>/kits.yml`.
- `game/GameSession.java` — `chosenKit` (id|none|random), селектор в `join`,
  переписан `giveKit` (системка + золото/предметы каста через `resolveKit`),
  очистка в `cleanup`.
- `listener/GameListener.java` — клик по селектору `kit-select` в лобби.
- `command/EscapeCommand.java` — `/escape kit list|copy|create|delete|edit|name|
  icon|gold|default` + tab + help.
- `util/Keys.java` — `KIT_ID`. `EscapePlugin` — wiring KitRegistry (load/reload).
- `messages.yml` — секция `kit.*` + строки в help.admin.

## Грабли, на которые наступил

  LEARNINGS: `<gold><gold>` схлопывался). Переименовал в `goldval`. Урок закреплён.

# 2026-07-20 (позже) — Мусорщик (скупка сломанного)

## Что сделано
NPC, которому продают сломанное снаряжение за золото — замыкает петлю экономики
(ломается → чинить/сдавать → золото). Согласовано в чате, записано в §14.

### Дизайн (со слов владельца)
- **Цена растёт с износом**: `цена = max(1, round(база(материал) × доля_износа))
  × количество`. Чем ушатаннее — тем больше (металлолом). Годное платит копейки.
- **Принимает только изношенное**: прочность есть, износ ≥ `min-wear-percent`
  (дефолт 25), материал в прайсе.

### Код
- `trader/TraderType.java` — секция `scrap` (Map<Material,Integer> + порог),
  `isScavenger()`, load/save. Третья роль рядом с shop/overseer.
- `menu/ScavengerMenu.java` — click-to-sell: сканирует инвентарь, показывает
  сломанные вещи с ценой (PDC `SCRAP_SLOT` = слот инвентаря), клик = продать одну,
  «Сдать весь хлам» = всё. Предмет берётся из инвентаря → потерять нельзя.
  Guard `hasPluginData` отсекает системку (SPECIAL_ITEM/CONTRACT_ID/THEME_ID/
  RESPAWN_OWNER/MARKER_TYPE). Звук наковальни на продажу.
- `menu/NpcMenu.java` — переписан на динамические кнопки (до 3 ролей),
  центрирование 1/2/3 кнопок, маршрут на ScavengerMenu.
- `listener/GameListener.java` — открытие NPC: roles>=2 → NpcMenu, иначе прямой
  вход (в т.ч. scavenger-only без «назад»).
- `command/EscapeCommand.java` — addscrap (материал из руки + база), removescrap,
  scrapwear (%), scraplist + tab + help.
- `arena/ArenaCheck.java` — scavenger теперь валидная роль (не даёт WARNING).
- `util/Keys.java` — `SCRAP_SLOT`. `messages.yml` — секции `scavenger.*`,
  `npc.scavenger-button`, `admin.scrap-*` + help.admin.
- `docs/examples/traders-prison.yml` — пример типа `МУСОРЩИК`.

### Решения по объёму
- **Стата продаж не добавлялась**: `StatsRepository.COLUMNS` — whitelist, новая
  колонка = ALTER TABLE + миграция существующей stats.db + правки Row/SELECT/UI.
  Владелец не просил; отложено (только DebugLog фиксирует продажи).

## Проверка

  проверено (см. PLANS).

# 2026-07-20 (ещё позже) — Модификаторы сессии («Режим дня»)

## Что сделано
Слой уникальности каждого матча БЕЗ новой сложности для игрока (обсуждали в
чате — владельцу «пипееееец» зашло). Согласовано, записано в §15.

### Дизайн
- Каждый матч случайно берёт 1 модификатор-кандидата из `modifiers.yml`.
- В лобби — **бюллетень** (предмет-спец `modifier-vote`, слот 4, ПКМ переключает
  голос). Владелец попросил: **если «за» > 1/3 стартующих — применяется**
  (`голосаЗа × 3 > числоСтартующих`).
- Эффект = множители/добавки к УЖЕ существующим числам, не новая механика.

### Код
- `modifier/Modifier.java` — id/name/desc/icon + Map<String,Double> effects,
  helpers mult()/add()/flag(). `modifier/ModifierRegistry.java` — глобальный
  `modifiers.yml` (read-only, random()). `resources/modifiers.yml` — 7 режимов.
- `EscapePlugin` — wiring (load/reload/accessor `modifiers()`).
- `GameSession`:
  - поле `candidateModifier` (ролл в конструкторе), `modifierVotes`, `activeModifier`.
  - бюллетень в `join` + `refreshBallots` (счётчик «нужно N» зависит от размера
    лобби), `toggleModifierVote`, `decideModifier(world)` в `startMatch` ДО
    генерации/раздачи, очистка в `cleanup`.
  - аксессоры `modMult/modAdd/modFlag`.
  - **эффект-хуки**: applyWear (`wear-add`), placeChests (`chest-count-mult`),
    giveKit (`start-gold-mult`), tick — интервал событий (`event-interval-mult`)
    и зарплата (`salary-mult`), refill (`refill-mult`), night-start (world.setTime).
- `listener/GameListener.java` — ПКМ по бюллетеню → `toggleModifierVote`.
- `menu/ScavengerMenu.java` — выплата × `session.modMult("scrap-mult")`.
- `messages.yml` — секция `modifier.*`.

### Детали
- Порог считается по игрокам, которые реально стартуют: `countModifierVotes()` =
  пересечение голосов с текущим `lobby` (голоса вышедших не в счёт автоматически).
- Ключи эффектов известны коду; новые модификаторы = только правка yml.
- night-start ставит время один раз (при ADVANCE_TIME=on ночь не вечная — как у
  Кровавой луны; атмосфера).

## Проверка
- `./gradlew build` — BUILD SUCCESSFUL. Смоук не делался. Живьём не проверено.

# Категории сундуков + интерактивный мастер настройки (§16)

## Проблема (со слов владельца)
Сундуки появлялись на ВСЕХ точках → игроки заучивали позиции и шли готовым
маршрутом. (Отбор случайного подмножества `chest-count` в коде уже был, но при
точках <= chest-count появлялись все.)

## Решение (согласовано в чате, записано в docs/07-decisions.md §16)
- Владелец выбрал: отбор = **квоты по категориям**; хранение = **как касты**
  (глобальная библиотека + копия на арену).
- Категория: `loot-min/max` (число слотов, заменяет 2-4), `quota` (сундуков за
  матч, крутится `chest-count-mult`), `refill-seconds` (0 = без рефилла). Пул
  лута — общий на арену (категория качество не меняет).
- Отбор: из точек каждой категории случайно `min(round(quota*mult), точек)`.
  Нет категорий у арены → прежнее поведение (одно подмножество `chest-count`).

## Код
Новые файлы:
- `category/ChestCategory.java` — модель (зеркало kit/Kit): load/save/copy/rollItemCount.
- `category/CategoryRegistry.java` — глобальный chest-categories.yml (зеркало KitRegistry).
- `resources/chest-categories.yml` — 4 шаблона (default/common/rare/hotspot).
- `arena/WizardState.java` — состояние мастера (арена + снимок точек + индекс).
- `arena/ChestSetupManager.java` — мастер: start/assign/advance/goTo/jumpTo/openGui/
  stop/onQuit/stopAll. Стеш инвентаря через PlayerSnapshot (крашоустойчиво: onJoin).
- `menu/ChestSetupMenu.java` — пагинированный выбор точки (клик = прыжок).
Правки:
- `arena/Arena.java` — chestSpots: List<Location> -> Map<Location,String> (точка->id
  категории, зеркало traderSpots); поле chestCategories; ОБРАТНО-СОВМЕСТИМЫЙ
  readChestSpots (старый плоский список -> default); load/save chest-categories.yml.
- `game/GameSession.java` — поле activeChestCategory; placeChests переписан на квоты
  + placeChestsLegacy (fallback); generateChestLoot — число слотов по категории;
  scheduleRefillIfEmpty — рефилл по категории (0 = не планировать); cleanup чистит.
- `arena/SetupMarkers.java` — call-site под Map (keySet/containsKey).
- `listener/SetupListener.java` — точка сундука = default; onWizardInteract (ПКМ
  жезлом), onWizardDrop (запрет дропа), onWizardQuit.
- `command/EscapeCommand.java` — subs chestcat/chestsetup + хендлеры + tab.
- `arena/ArenaCheck.java` — quota-ветка (сироты/перекос) + summary-chests-quota.
- `EscapePlugin.java` — wiring реестра+менеджера, reload/onDisable stopAll, guard в
  joinArena (isActive/isArenaBusy), аксессоры categories()/chestSetup().
- `util/Keys.java` — CATEGORY_ID. `messages.yml` — chestcat.*/chestsetup.*/check.msg.cat-*/help.

## Грабли / решения
- Миграция zero-touch: старый `chests:` (плоский список) грузится, точки -> default;
  пере-сейв апгрейдит формат. Арена без категорий = legacy, риск нулевой.
- Выход/краш в мастере: снапшот на диске в onQuit НЕ удаляем — восстановит
  GameListener.onJoin (тот же путь, что после краша). Явный stop = restore+delete.
  Guard joinArena: снапшот мастера не затрётся входом; арена под мастером не пускает.
- Сирота-точка (категория удалена/не скопирована, напр. старый `default` при
  скопированных только rare/common) в квота-режиме НЕ появляется; ArenaCheck
  предупреждает (cat-orphan-points/cat-no-points). Чинится мастером.
- Жезл вне матча — в SetupListener (GameListener.onInteract выходит рано без сессии).
