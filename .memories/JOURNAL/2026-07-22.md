# 2026-07-22 — Редизайн лута/сундуков (§17)

Полная замена системы лута со слов владельца. Спека утверждена поштучно
(14 edge-кейсов + контракты + рефилл), лежит в `PLANS.md` §17.

## Что сделано

**Модель.** Категория лута теперь ГЛОБАЛЬНАЯ (`loot/<id>.yml`), несёт И лут, И все
лимиты (вес 1–1000, min/max-per-chest, min/max-per-arena, min/max-chests,
refill-seconds). Пер-ареновый `loot.yml` и пер-ареновые `chest-categories.yml`
УБРАНЫ. Точка сундука несёт СПИСОК id категорий (0..N), формат `locations.yml`
`{location, categories:[...]}` (бэк-компат чтения одиночного `category`/плоского
Location). Контракты — свой конфиг в `arena.yml` (`contract-min/max-per-arena`,
`contract-min/max-per-chest`), кладутся независимо от лута.

**Спавн (GameSession, 3 фазы):** назначение точке ≤1 категории (Проход 1 —
минимумы по убыванию веса; Проход 2 — добор взвешенно; Проход 3 — филлеры
`max=-1`), все точки сперва воздух, выбранные → CHEST + лут. Бюджет
`max-per-arena` мягкий (`min-per-chest` жёсткий, кейс 10). Рефилл per-category,
аддитивный (докидывает в свободные слоты, старое лежит), триггер по первому
забору. Динамические сундуки — категория с наибольшим весом. `giveRandomLoot` —
взвешенный выбор глобальной категории.

**GUI** `/escape loot [gui]` — экран категорий (список + создать из
инвентаря/сундука/копией + удалить) → редактор категории (настройки кликом,
сетка предметов). Текст-команды `/escape loot list|create|delete|name|icon|weight|
minchest|maxchest|minarena|maxarena|minchests|maxchests|refill`. Мастер-жезл
`/escape chestsetup` — ПКМ ТОГГЛИТ категорию на точке (мультивыбор), стрела =
следующая точка (`NEXT_TAG`). `/escape chesttag` — глобальные метки, тоггл.

**Миграция** (`loot/LootMigration`, одноразовая, маркер `loot/.migrated`):
старые `arenas/<id>/loot.yml`+`chest-categories.yml` → глобальные
`<arena>_<oldcat>` с пулом арены как лутом; точки ремапятся. Плоские старые
точки (в старой модели = «default») привязываются к `<arena>_default`, если та
категория есть (иначе остаются без категории).

## Как делалось

Ядро (LootCategory, LootCategoryRegistry, Arena, EscapePlugin) — вручную с
зафиксированным API. Остальное — 5 параллельных субагентов по непересекающимся
файлам против контракта: A=GameSession, B=GUI(+5 меню-классов), C=миграция,
D=мастер-жезл, E=SetupListener+ArenaCheck. Координатор (EscapeCommand,
messages.yml, NEXT_TAG-роутинг, straggler в ArenaManager) — вручную. Все ключи
игрокам собраны из отчётов агентов в `messages.yml`.

## Дефект по ходу и фикс

Первый прогон миграции: плоские старые точки CLASSIC (42) стали ПУСТЫМИ вместо
`classic_default` (мой новый `readChestSpots` даёт плоской точке пустой список, а
`defaultId` не выставлялся, когда у арены была явная старая категория «default»).
Фикс в `LootMigration`: (1) старая категория с id «default» запоминается как
`defaultId`; (2) пустой список точки → `defaultId`, если он есть. Перепроверено
сбросом состояния (возврат старых id + `rm loot/`) и повторным смоуком — 0 пустых
на CLASSIC.

## Правка после плейтеста: СОВМЕЩЕНИЕ категорий (не выбор одной)

Владелец поиграл: если у сундука несколько категорий, выбор ОДНОЙ случайной давал
«в редком сундуке одна вещь» (редкая категория диктует min/max-per-chest=1). Новое
поведение — **совмещать**: сундук наполняется лутом из ВСЕХ своих категорий, редкая
добавляет предмет ПОВЕРХ базовой.

Изменения только в `GameSession` (модель упростилась — 3-фазное присвоение одной
категории убрано):
- `activeChestCategory: Map<Location,String>` → `activeChestCategories: Map<Location,List<String>>`.
- `placeChests`: каждая категория НЕЗАВИСИМО берёт свои сундуки в [min,max]-chests из
  своих точек (max=-1 → во всех своих точках); точка, выбранная несколькими → совмещение.
- `fillChestInitial`/`fillChestRefill` принимают `List<LootCategory>`, наполняют по весу
  DESC (весомая гарантирует свой per-chest минимум первой, когда слоты кончаются, cap 27).
  Бюджет max-per-arena по-прежнему per-category и мягкий.
- Рефилл: таймер сундука = минимальный положительный `refill-seconds` среди его категорий;
  на срабатывании докидывает из всех.
- Динамический сундук — по-прежнему одна категория (наибольший вес), но хранится списком.
- `weightedPickCategory` остался только для `giveRandomLoot` (награда события).
- Вес категории теперь: (1) порядок наполнения в совмещённом сундуке, (2) выбор
  категории динамического сундука и награды. Для присвоения точкам он больше НЕ нужен.

Собрано (build зелёный), задеплоено, смоук чист. Логика совмещения — на плейтест.

## Правка 2: контракты не видны в сундуках

Владелец: контрактов в сундуках нет. Диагностика: механизм раздачи рабочий
(тот же паттерн записи, что у лута/рефилла), но раздача была **пер-арена** —
клала по 1 контракту на сундук за проход, пока не наберёт `max-per-arena`. При
дефолте `max-per-arena: 6` над сотней сундуков — призрак; даже при 36 «по одному»
почти не видно.

Фикс: раздача переписана на **ПЕР-СУНДУК как основу** — каждый сундук катает
`[min,max]-per-chest`; `max-per-arena` теперь МЯГКИЙ потолок и поддерживает
`-1` = без потолка (дефолт сменён `6` → `-1`, тогда плотность целиком задаёт
per-chest). `min-per-arena` — пол (добор, если недобрали). В `placeOneContract`
убран ошибочный guard `free >= 27` (отсекал верхнюю половину двойных сундуков).
Существующим аренам с сохранённым `contract-max-per-arena: 36` — либо оставить
как потолок, либо `/escape set <ID> contractmaxarena -1`. Собрано, смоук чист.
Раздача проверяется на матче (`/escape debuglog on` → строка `contracts-placed
... placed=N`).
