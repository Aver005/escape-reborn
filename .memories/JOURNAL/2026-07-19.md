## Что вскрылось попутно

- **[BUG] `ShopMenu` вешает сервер при >28 трейдов** — бесконечный `while` на
  main thread. Записан в `BUGS.md`, не исправлен.
- **Лимит 7 темок на NPC** (`ThemesMenu:66`) — тоже в `BUGS.md`.
- **Таймеры считаются по остатку `remaining`**, а не по прошедшему времени:
  число срабатываний = `floor((duration - 1) / interval)`. На дефолтах
  (1200/600) зарплата выдаётся **ровно один раз за матч** — вероятно, не то, что
  задумывалось. Требует решения владельца.
- **Дефолты дают WARNING валидатора**: `glowtime 600` при `duration 1200`
  попадает под правило «glow >= duration/2» (`ArenaCheck:105`). Каждая свежая
  арена создаётся с предупреждением.
- **`defaults` в `config.yml` не покрывает** `wear-min-percent`,
  `wear-max-percent`, `dynamic-chests` — `ArenaManager.applyDefaults` их не
  читает, новые арены всегда получают 40/90/false из полей `Arena`.
- **`fork-uses`**: `config.yml` = 4, но фолбэк `Arena.load` = 1. Расходится для
  рукописных `arena.yml` без ключа.
- **ID контрактов/темок/NPC регистрозависимы** (реестры ищут по точному ключу),
  а команды аплоуеркейсят аргумент → сущность с ID строчными буквами загрузится,
  но станет недоступна командам. ID арен, наоборот, нечувствительны.
- **Комментарий в `config.yml:37`** («Оффлайн-страж») стоит над секцией `match:`,
  а не над `offline-guard:` — смещён, путает.

## Фикс ShopMenu (по запросу владельца, тем же днём)

`menu/ShopMenu.java` — граница внесена в условие цикла:
`while (slot < inventory.getSize() && isBorder(slot))`. Ёмкости вынесены в
константы `ShopMenu.MAX_TRADES = 28` и `ThemesMenu.MAX_THEMES = 7`.
`ArenaCheck` получил два WARNING (`npc-trades-overflow`, `npc-themes-overflow`)
+ ключи в `messages.yml`, чтобы лишний контент не пропадал молча.

**Как проверялось:**

1. Симуляция раскладки слотов (`scratchpad/SlotSim.java`) для 0..200 товаров:
   старый цикл виснет ровно на 29-м, новый всегда завершается и кладёт
   `min(n, 28)`. Для n <= 28 старый и новый дают одно и то же — регрессии нет.
2. `./gradlew build` + `deploy` + смоук на `E:\Servers\escape`:
   `[Escape] Enabling`, `Arena loaded: CLASSIC`, `Done (12.022s)`, чистая
   выгрузка, 0 стектрейсов (единственный WARN — сторонний oshi-счётчик Windows,
   был и до фикса).
3. Ключи `npc-*-overflow` подтверждены внутри задеплоенного jar.

**Не проверено вживую:** само открытие магазина с 29 трейдами и срабатывание
новых WARNING в `/escape check` — нужен клиент и игрок, командой это не
воспроизвести (`/escape` работает только от игрока).

**Грабля инфраструктуры:** `start.bat` через `cmd /c` из фонового bash-таска
молча не поднимает сервер (пустой вывод, лог не обновляется) — легко принять
старый `latest.log` за свежий смоук. Рабочий способ:
`(sleep N; echo stop) | java -Xms2G -Xmx4G -jar paper.jar --nogui` — stdin живёт,
сервер останавливается штатно.

## Блоки-подсказки настроенных точек арены (`SetupMarkers`)

**Задача владельца:** при размещении сундука/спавна/торговца на арене не
появлялось никаких блоков — не видно, где что уже размещено.

**Сделано** (`arena/SetupMarkers.java` — новый класс, вся логика подсказок):

| Точка | Подсказка | Слом |
|---|---|---|
| спавн | 2 серых стекла (фигура игрока) | точка удаляется |
| финальный спавн | 2 чёрных стекла | точка удаляется |
| точка NPC | 2 розовых стекла | точка удаляется |
| точка сундука | настоящий сундук | точка удаляется |

- Стёкла ставятся колонной в 2 блока ровно там, где встанет фигура.
- Сундук-точка остаётся блоком карты; его содержимое **не участвует в игре и
  чистится всегда** — в `clearForMatch` (старт) и в `placeAll` (после матча).
- Слом любого из двух стёкол убирает точку и второй блок; чинится в
  `SetupListener.onBreak` (только `escape.admin`, только когда `arena.getSession()
  == null` и ломающий не в сессии — участников матча ведёт `GameListener`).
  `setDropItems(false)`, содержимое сундука-точки чистится до слома.
- Жизненный цикл: `GameSession.startMatch` → `clearForMatch(arena)` **до**
  `placeChests()` и телепорта (игрок не должен появиться внутри стекла);
  `cleanup()` → `placeAll(arena)` **после** отката `editedBlocks`.
- `/escape markers <ID>` — расставить подсказки заново (для арен, настроенных
  раньше). Отказ, если арена в игре (`admin.markers-busy`). Идемпотентна.
- Маркер встаёт **только в воздух/заменяемый блок** (`isFree`) — карту не
  затирает; занятая точка пропускается.
- `BlockPlaceEvent` для маркеров по-прежнему отменяется (маркер не тратится),
  поэтому подсказка ставится **следующим тиком** (`runTask`): отмена события
  восстанавливает блок и затёрла бы её.
- Новые ключи: `admin.marker-removed`, `admin.markers-placed`,
  `admin.markers-busy` + 3 строки в `help.admin`. Дефолты берутся из jar, ручное
  слияние `messages.yml` не требуется.

**Тонкость сундуков:** точки, попавшие в матч, и так перезаписываются
`placeChests()`, а `editedBlocks` запоминает CHEST → `cleanup` восстанавливает
сундук-подсказку сам. Точки вне выборки просто стоят пустыми.

**Проверено:** `./gradlew build` + `deploy` + смоук на `E:\Servers\escape`:
`[Escape] Enabling`, `Arena loaded: CLASSIC`, `Done (11.999s)`, штатная
остановка через `(sleep 45; echo stop) | java ...`, 0 стектрейсов/ERROR.

**Не проверено вживую:** сами блоки в мире (постановка, снятие на старте,
возврат после матча, удаление точки сломом, `/escape markers`) — нужен игрок в
игре, `/escape` работает только от игрока. Это первый пункт плейтеста.

**Обновлены доки:** `docs/guides/01-commands.md` (таблица маркеров + раздел
«Блоки-подсказки точек» + раздел про `/escape markers`),
`03-arena-create.md` (спавны/сундуки/NPC/финал), `06-operations.md`.

## Улучшенные ванильные механики: снежки/яйца и удочка-крюк

Запрос владельца: сделать так, чтобы в матче снежки и яйца наносили урон
(яйца сильнее, но убить можно только «затарившись»), а удочкой можно было
подтягивать игроков — как хук Пуджа, с силой от зачарования «Сила» на удочке.
Решение записано в `docs/07-decisions.md` §12.

**Новый листенер** `listener/MechanicsListener.java` — вся «улучшенная ваниль»
в одном месте; вне сессии не вмешивается вообще.

### Снежки и яйца

- Ванильно оба бьют игрока на **0 урона**, поэтому урон выдаём сами.
- **Ключевой выбор — `ProjectileHitEvent`, а не правка `EntityDamageByEntityEvent`.**
  Причина: `Player#hurt` в ванилле имеет ветку `f == 0 → return false` и
  ветку «в окне неуязвимости и `amount <= lastHurt` → return false» — то есть на
  событие с 0 урона полагаться нельзя, оно может вообще не долететь до плагинов.
  `ProjectileHitEvent` же приходит **до** ванильного попадания: мы наносим урон
  первыми, ванильные 0 после этого гасятся выставленным окном неуязвимости —
  задвоения нет by design.
- Урон через `DamageSource.builder(DamageType.THROWN).withCausingEntity(стрелок)
  .withDirectEntity(снаряд)`: работает броня, `getDamager()` = снаряд, а наш
  `GameListener.resolveDamager` уже умеет разворачивать `Projectile → shooter`.
  Бесплатно получили множитель «Кровавой луны», кредит убийце и фейковую смерть.
- Балансу помогает ванильное окно неуязвимости (1 сек): спам в упор
  **не складывается**, засчитывается один снаряд в секунду.
- Дефолты: снежок 2.0 (сердце), яйцо 3.0 (1.5 сердца) — `config.yml` →
  `projectiles`. 0 = вернуть ванильное поведение.
- **Куры.** Раз яйцо стало оружием, их будут кидать пачками, а вылупившиеся
  куры остались бы на карте (в `spawnedEntities` их никто не регистрирует —
  инвариант «мир как до матча» нарушился бы). `ProtectionListener` теперь
  режет `CreatureSpawnEvent` с `SpawnReason.EGG` в мирах арен.

### Удочка-крюк

- `PlayerFishEvent` + `State.CAUGHT_ENTITY`, цель — игрок той же сессии.
- Сила = `base + per-level * getEnchantmentLevel(Enchantment.POWER)`, потолок
  `max-strength`. Дефолты 0.55 / 0.30 / 2.5: без зачарования рывок на пару
  блоков, «Сила V» ≈ 2.05 — полёт через полкарты.
- Вектор нормализуется и получает дугу (`y * 0.5 + 0.35`), иначе жертву
  просто тащит по земле.
- **Грабли: событие приходит ДО ванильного `pullEntity`**, который добавляет
  свои `(рыбак − крючок) * 0.1` к скорости. Поэтому свою скорость выставляем
  **следующим тиком** (`runTask`) — так наше значение перекрывает ванильное
  целиком и рывок предсказуем. Отменять событие нельзя: при отмене ванильный
  `retrieve()` выходит до `discard()` и крючок повисает в мире навсегда.
- `session.recordDamager(жертва, рыбак)` — разбился после крюка в пределах
  обычного окна кредита, значит убийство рыбака.
- Удочка берётся из руки по `e.getHand()` (может быть null → основная рука).

### Как получить удочку с «Силой»

Ванильно никак — ни стол, ни наковальня. Только админ: `enchants: {power: N}`
в `loot.yml`/`traders.yml` (рукописный формат уже это умеет) либо
`/give <ник> fishing_rod[enchantments={power:N}]` (админам команды в матче
не режутся — `ChatListener` пропускает `escape.admin`).
